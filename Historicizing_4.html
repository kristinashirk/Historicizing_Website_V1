Web Accessibility

The Web is fundamentally designed to work for all people, whatever their hardware, software, language, culture, location, or physical or mental ability. When the Web meets this goal, it is accessible to people with a diverse range of hearing, movement, sight, and cognitive ability (Accessibility, 2018).

Central to accessible websites are a solid information structure and rich navigation elements. Navigation elements allow users using screen readers to choose whatever way of finding pages is easiest for them. Users with low vision may find using search easier than navigating through a large menu. Users with cognitive impairments may prefer a table of contents or site map over clicking through many pages (Navigation, 2018). Web accessibility began as an afterthought to web design. It was an optional add-on by charitable designers. Today, however, web accessibility is a crucial aspect of web design. The following discusses the evolution of accessible web navigation and the implications of the nav element in modern web design.

ARIA and Accessibility

A precursor to HTML5 and the nav element was ARIA. ARIA, or Accessible Rich Internet Applications, is a set of attributes that define ways to make Web content and Web applications more accessible to people with disabilities (ARIA, 2018).

ARIA supplements HTML so that interactions and widgets commonly used in applications can be interpreted by accessible technologies. For example, ARIA enables accessible navigation landmarks in HTML4, JavaScript widgets, form hints and error messages, live content updates, and more (ARIA, 2018).

Navigation in ARIA is defined by navigation landmark roles. The ARIA navigation role works much like the modern nav element, by providing a way to identify the organization and structure of a web page (ARIA: Navigation Role, 2018). By using the navigation roles to classify and label sections of a page, structural information is conveyed visually through layout and is represented programmatically (ARIA: Navigation Role, 2018). These structural cues allow screen readers to provide keyboard navigation to important sections of a page (ARIA: Navigation Role, 2018).

Nav and Accessibility

However, many of the supplemental mechanics of ARIA were incorporated into HTML5, including the nav element.

As mentioned above, the nav element works in much the same way as ARIA navigation roles - it marks content contained within the tag as navigational elements. Browsers, such as screen readers for disabled users, can use the nav tag to determine whether to omit the initial rendering of this content (HTML nav tag, 2018).

The nav element has the advantage over using ARIA supplemental roles in that using the nav element will automatically communicate a section has a role of navigation (ARIA: Navigation Role, 2018). The labeling benefits of ARIA navigation roles are embedded in the nav element.

It is advised by accessibility experts that developers use the HTML element over using ARIA attributes (ARIA: Navigation Role, 2018).

Ways to improve your site’s navigation for accessibility:

Stephanie Lin of the UXBooth presents the following rules for creating effective, accessible website navigation in her article The Rules for Modern Navigation:

Prioritize consistency

  • Consistently employ navigational elements throughout your site. For example, if your main navigation bar includes dropdown menus displaying subpage links, each element in the main bar should have a dropdown menu.
  • Use secondary navigational elements consistently throughout the site. This reinforces your site’s information hierarchy and helps the user easily navigate from one page to another.
  • Ensure all navigational elements function and link to a landing page.
  • Visual design should clearly indicate the function of a navigational element, and whether that element is a link or simply a design element.
  • Use clear language in your link descriptions so users know exactly where the link leads to. This is especially important for users who rely on screen readers.

Design clear interactions

  • When a site visitor doesn’t know which items are interactive, or what interactions to expect, confusion results. Combat this confusion by thoughtfully designing interactions to be clear and understandable.
  • Use visual change to indicate interactions. The change clearly shows users when an element has been used, or if it is an interactive element.
  • Use labels. Labels clearly indicate the function of an element and reduce ambiguity. Labels are especially important to users using screen readers and other accessibility tools.

Avoid deep navigation

  • Strive for a “flat navigation,” where the user can access the deepest nested page in the site within one to two clicks. This may require you to reconstruct your site's hierarchy, reclassify and consolidate information, or remove content to reduce the complexity of your site.
  • Use short menus. Short menus reduce the congnitive load for users and make it easier for them to locate the information they need.
  • Visually design your menu to mimic the information hierarchy of your site. Use font size, color, and weights to indicate the underlying structure. This will allow users to quickly understand which links are primary, secondary, and tertiary navigation items.
  • Use location indicators, like breadcrumbs, to show users where they are within your site.

Design for responsive compatibility

  • Remove hover interactions in your main navigation elements to avoid confusion for users, make screen reader navigation easier, and make your site easily adaptable to mobile.
  • Use different navigation for mobile and desktop. Optimizing navigation for each device will make the overall user experience of your site much better.
