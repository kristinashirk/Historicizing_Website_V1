Web Accessibility

The Web is fundamentally designed to work for all people, whatever their hardware, software, language, culture, location, or physical or mental ability. When the Web meets this goal, it is accessible to people with a diverse range of hearing, movement, sight, and cognitive ability.

Central to accessible websites is a solid information structure and rich navigation elements. Navigation elements allow users using screen readers to choose whatever way of finding pages is easiest for them. Users with low vision may find using search easier than navigating through a large menu. Users with cognitive impairments may prefer a table of contents or site map over clicking through many pages. Web accessability began as an afterthought to web design. It was an optional add-on by cheritable designers. Today, however, web accessability is a crutial aspect of web design. The following discusses the evolution of accessible web navigation and the implications of the nav element in modern web design.

ARIA and Accessibility

A precursor to HTML5 and the nav element was ARIA. ARIA, or Accessible Rich Internet Applications, is a set of attributes that define ways to make Web content and Web applications more accessible to people with disabilities().

ARIA supplements HTML so that interactions and widgets commonly used in applications be interpreted by accessible technologies. For example, ARIA enables accessible navigation landmarks in HTML4, JavaScript widgets, form hints and error messages, live content updates, and more” ().

Navigation in ARIA is defined by navigation landmark roles. The ARIA navigation role works much like the modern nav tag, by providing a way to identify the organization and structure of a web page. By using the navigation roles to classify and label sections of a page, structural information is conveyed visually through layout is represented programmatically. These structural cues allow screen readers to provide keyboard navigation to important sections of a page.

Nav and Accessibility

However, many of the supplemental mechanics of ARIA were incorporated into HTML5, including the nav element.

As mentioned above, the nav element works in much the same way as ARIA navigation roles - it marks content contained within the tag as navigational elements. Browsers, such as screen readers for disabled users, can use the nav tag to determine whether to omit the initial rendering of this content.

The nav element has the advantage over using ARIA supplemental roles in that Using the nav element will automatically communicate a section has a role of navigation. The labeling benefits of ARIA navigation roles are embedded in the nav element.

It is advised by accessibility experts that developers use the HTML element over using ARIA attributes ().
